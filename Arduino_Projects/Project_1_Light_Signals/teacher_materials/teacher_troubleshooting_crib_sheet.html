Teacher Troubleshooting Crib Sheet

Project 1 • Light Signals • Internal teacher reference
Eight most common failure modes in Project 1, in rough frequency order, with the fix. Use this as a quick reference during rotation when a student is stuck and the R2 stuck protocol did not resolve it. The symptom is what the student reports; the cause is what is usually actually wrong; the fix is the first thing to try.

1. LED does not light at Milestone 4 (or later milestones with a new LED).

Usual cause: LED wired backwards. The long leg must go to the resistor and then to pin 9 (or pin 10 or pin 11); the short leg goes to GND. A student wiring the LED the wrong way round is the single most common Milestone 4 failure.

Fix: flip the LED. If that does not fix it, check that the 220 Ω resistor is actually in series with the LED (between the pin and the LED's long leg), not just stuck uselessly in the breadboard.

[Verification: Principle 3 (physical-first) + R1 wiring reference card. Berrezueta-Guzmán et al. (2021) non-human-mediator framing — the reference card catches ~60% of these before the teacher needs to rotate.]

2. Arduino IDE shows "Arduino not found" or "no COM port" on Upload.

Usual cause: wrong COM port selected in Tools → Port, or the USB cable is a "charge-only" cable with no data lines.

Fix: scroll through the available ports in Tools → Port and pick the one that appeared when the Arduino was plugged in. If no new port appears when the Arduino is plugged in, swap the USB cable for a known-good data cable from the parts library.

3. Upload says "avrdude: stk500_getsync() attempt X of Y" error.

Usual cause: another program has the COM port locked — usually a Serial Monitor window or another Arduino IDE instance left open from a previous session.

Fix: close the Serial Monitor, close any extra IDE windows, try Upload again. If it still fails, unplug the USB cable, wait three seconds, plug it back in, retry.

4. Button does not do anything at Milestone 8.

Usual cause: 10 kΩ pull-down resistor on the wrong side of the button. The resistor must be between pin 2 and GND, not between 5 V and GND. A student wiring the pull-down between 5 V and GND creates a persistent short circuit whenever the button is pressed.

Fix: move the 10 kΩ resistor so it sits between pin 2 and GND. See R1 wiring reference card, Wiring 3 diagram.

[Verification: Principle 3 — physical-first. The wiring reference card (R1) carries the correct layout, and the teacher-rotation checkpoint at Milestone 7 is specifically designed to catch this before the student gets discouraged.]

5. Both LEDs stay on all the time after Milestone 6 (alternating sketch).

Usual cause: one of the LEDs is wired to 5 V instead of to a digital pin. Both LEDs' long legs should go through their own 220 Ω resistors to pin 9 and pin 10 — not to the 5 V rail.

Fix: re-check wiring against R1 Wiring 2 diagram. Move any jumper wire that is going from an LED to 5 V so that it goes to the correct digital pin instead.

6. Arduino resets or turns off repeatedly.

Usual cause: short circuit somewhere in the wiring — usually a jumper wire accidentally bridging 5 V to GND, or (at Milestone 7+) a button wired without the pull-down resistor so that pressing the button shorts 5 V to GND.

Fix: disconnect the USB cable. Look for any wire going directly between 5 V and GND (both sides of the breadboard's power rails). Remove the short. Re-check the pull-down resistor position if the button is wired. Reconnect.

7. Student says "my sketch is the same as the reference but it does not work."

Usual cause: the student opened the sketch in the IDE but did not click Upload, or Upload silently failed with a compile error the student did not notice.

Fix: click Upload and watch for the green "Done uploading" message at the bottom of the IDE. If there is no green message, read the red error text. If the error is "avrdude: stk500_getsync()" see item 3.

8. Student's LED lit up brightly for a moment then stopped working.

Usual cause: LED wired without a current-limiting resistor, drew too much current, burned out. This is the most common way to destroy an LED in Project 1.

Fix: replace the LED from the parts library. Add the 220 Ω resistor. Re-wire following R1 Wiring 1. Use this as a teaching moment — "the resistor is not optional, this is why."

[Verification: Principle 3 + R4 safety reminder card — the safety card addresses this directly as Rule 2.]

9 (digital infrastructure). Claude Code does not see the student's sketch files, or opens pointed at the wrong folder.

Usual cause: Claude Code was left open from a previous student's session with their working directory, and the new student is seeing the previous student's folder. Or: Google Drive for Desktop is not fully synced and the student's folder has not yet appeared locally.

Fix: close Claude Code completely, verify Drive for Desktop is running and synced (open File Explorer and navigate to the student's folder manually — if it appears, Drive is synced), then relaunch Claude Code pointed at G:\My Drive\Arduino_Projects\<nickname>\Project_1_Light_Signals\.

[Verification: Arduino_PBL_Program.md §9.5 item 4 — three-way digital-infrastructure dependency. This is an expected failure mode; note any occurrence in the Weekly Data Collector (§7.3 Instrument 2).]
